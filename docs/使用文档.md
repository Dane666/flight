# 机票监控使用文档

## 1. 功能说明

本项目用于监控指定航线与日期窗口内的往返机票价格，支持：

- 网页抓取真实价格（`provider: trip_scrape`）
- Google Flights 结构化查询（`provider: google_flights`）
- 演示价格（`provider: mock`）
- 控制台输出告警（`notifier: console`）
- 邮件通知（`notifier: email`，可选）
- 飞书机器人推送（`notifier: feishu`，可选）

当前默认测试规则：

- 出发地：`CAN`、`SZX`、`HKG`
- 目的地：`PQC`
- 日期：端午节前后 5 天窗口

如果你是第一次使用，建议直接先看：

- 第 3 节：怎么改自己的出发地、目的地、日期
- 第 4.3 节：怎么执行一次最低价汇总
- 第 5.3 节：Google Flights 当前能输出哪些字段

## 3. 配置文件

### 3.1 初始化默认配置

```bash
python main.py init-config --force
```

### 3.2 核心配置（`config.yaml`）

- `provider`: 数据源（推荐先保留 `trip_scrape`，需要结构化字段时可切到 `google_flights`）
- `serpapi_api_key`: Google Flights（SerpApi）API Key
- `currency`: 输出币种（建议 `CNY`）
- `google_flights_hl/google_flights_gl`: Google Flights 查询语言与地区
- `trip_scrape_timeout_seconds`: 抓取超时秒数
- `alert_threshold`: 告警阈值
- `window_start/window_end`: 监控日期窗口
- `origins`: 去程出发地列表，例如 `["CAN", "SZX", "HKG"]`
- `destination`: 主监控目的地，例如 `PQC`
- `thailand_destinations`: 泰国目的地列表，例如 `["BKK", "HKT"]`
- `fixed_depart_date/fixed_return_date`: 固定去返日期；若同时配置，`run-once` 会只查这一组，汇总命令会以此为中心扩窗匹配
- `min_trip_days`: 最小行程天数（默认 4）
- `max_trip_span_days`: 去返总跨度上限（默认 6 天，含端午假期）
- `max_leave_workdays`: 除端午假期外允许请假的工作日上限（默认 3 天）
- `feishu_webhook_url`: 飞书机器人 webhook 地址
- `feishu_secret`: 飞书签名密钥（可空）

补充说明：

- 系统不会简单地在窗口里任意找最低价，而是会额外要求往返日期完整覆盖端午假期
- 允许提前去程、延后返程，但不会返回未覆盖端午假期的日期组合
- 默认还要求总去返跨度不超过 6 天，且除端午假期外请假工作日不超过 3 天

### 3.3 第一次使用时最常改的 4 项

如果你只想先跑通一遍，通常只改下面几项：

```yaml
provider: google_flights
serpapi_api_key: "<YOUR_SERPAPI_KEY>"
origins:
  - HKG
destination: PQC
thailand_destinations:
  - BKK
window_start: "2026-06-17"
window_end: "2026-06-22"
notifier: console
```

含义：

- `origins`: 你从哪些城市出发
- `destination`: 你主关注的往返目的地
- `thailand_destinations`: 如果你还想顺便比较泰国最低价，就把候选城市放这里
- `window_start/window_end`: 允许系统在这个日期范围里帮你找最便宜的组合

### 3.4 如果你已经知道大致去返日期

可以配置：

```yaml
fixed_depart_date: "2026-06-19"
fixed_return_date: "2026-06-24"
```

说明：

- `run-once` 会直接按这组日期查询
- `run-best-deals-summary` 和 `run-thailand-cheapest` 会以这组日期为中心，前后约 5 天扩窗后再匹配
- 这样既保留“附近日期更便宜”的能力，也不会完全跑散

### 3.5 推荐配置（真实抓取 + 控制台）

```yaml
provider: trip_scrape
currency: CNY
notifier: console
```

说明：Trip 抓取原始价格通常是 USD，程序会自动换算为 `currency` 指定币种（如 CNY）。
当前 `trip_scrape` 已改为“两阶段抓取”：

- 第一阶段快速扫价格，优先压缩全窗口遍历时间
- 第二阶段只对最终最低价候选补抓详情，减少无效深抓

### 3.6 Google Flights（SerpApi）配置

```yaml
provider: google_flights
serpapi_api_key: "<YOUR_SERPAPI_KEY>"
google_flights_hl: en
google_flights_gl: hk
currency: USD
notifier: console
```

说明：

- 该模式是新增独立 provider，不会覆盖旧的 `trip_scrape`。
- 当前实现会先查询去程最低价候选，再使用 `departure_token` 补抓返程。
- 因此 `google_flights` 通常能拿到更完整的 `go/back` 时刻、航班号、往返路线、承运航司、舱位、机型、时长和价格参考。
- 如果 Google Flights 因配额耗尽、网络失败或 API 错误不可用，程序会在当前运行中自动切回 `trip_scrape` 继续检索。
- 建议先复制一份临时配置文件验证，再决定是否切换正式配置。

## 4. 运行方式

### 4.1 执行一次

```bash
python main.py run-once --config config.yaml
```

快速模式（推荐日常快速巡检）：

```bash
python main.py run-once --config config.yaml --quick
```

说明：`--quick` 会将每个出发地的抓取组合降到 1 条，显著缩短等待时间。

运行后会立即输出 `[RUN]` / `[SCRAPE]` 进度日志，避免“终端无响应”的观感。
如果你在某些终端里仍感觉输出延迟，可使用：

```bash
python -u main.py run-once --config config.yaml
```

### 4.2 持续运行

```bash
python main.py run --config config.yaml
```

### 4.3 飞书汇总推送

在 `config.yaml` 配置：

```yaml
notifier: feishu
feishu_webhook_url: "https://open.feishu.cn/open-apis/bot/v2/hook/xxxxx"
feishu_secret: null
```

执行：

```bash
python main.py run-best-deals-summary --config config.yaml
```

该命令会输出 `[DEAL-PQC]` 与 `[DEAL-TH]`，并将相同汇总推送到飞书。

如果当前数据源拿不到某个字段，飞书里会直接省略这一行，不再显示“待下单页确认”的占位文案。

## 5. 输出字段解释

控制台输出示例：

```text
[QUOTE] CAN->PQC 2026-06-18/2026-06-20 dep=16:20 arr=09:35+1d 🔥PRICE=2210.55 CNY🔥 (src=320.00 USD, fx=6.9080) position=低位 hist(min=2210.55, max=2210.55, avg=2210.55)
```

字段含义：

- `dep`: 去程起飞时间
- `arr`: 去程到达时间
- `PRICE`: 醒目标记的最终价格（换算后的目标币种）
- `src`: 源站抓到的原始价格与币种
- `fx`: 汇率（源币种 -> 目标币种）
- `position`: 在历史价格中的位置（`高位` / `中位` / `低位` / `新样本`）
- `hist`: 历史最小/最大/均值

### 5.1 检索三地到泰国最低价（窗口内匹配）

```bash
python main.py run-thailand-cheapest --config config.yaml
```

会按日期窗口内的去返组合扫描：

- 出发地：`origins`
- 泰国目的地：`thailand_destinations`

若配置了 `fixed_depart_date/fixed_return_date`，会在该日期段前后约 5 天自动扩窗后匹配。
同时仍要求去返日期完整覆盖端午假期。

最后打印一条 `[TH-CHEAPEST]` 结果。

### 5.2 只输出最终最低价（PQC + 泰国）

```bash
python main.py run-best-deals-summary --config config.yaml
```

该命令不会打印中间抓取日志，只输出两条：

- `[DEAL-PQC]`：三地到 PQC 的最低价
- `[DEAL-TH]`：三地到泰国目的地列表的最低价

PQC 与泰国会按各自最优去返日期独立匹配，不再强制同一组日期。
但两类结果都会满足“完整覆盖端午假期”的约束。

并包含：

- 去程时间：`go=起飞->到达`
- 返程时间：`back=起飞->到达`
- 返程路程：`back_route=`
- 返程中转站：`back_stop=`

### 5.3 Google Flights 当前可输出的字段

`provider: google_flights` 时，当前会尽量输出这些字段：

- 去程 / 返程时间
- 去程 / 返程路线
- 去程 / 返程中转站与中转明细
- 航班号
- 承运航司
- 舱位
- 机型
- Google 价格参考（如 `price_level`、`lowest_price`、`typical_price_range`）
- 碳排参考

说明：

- Google Flights 搜索结果本身可以稳定拿到上面这些结构化字段
- 行李和退改签理论上需要更后面的 booking options / 下单链路，但这条链路在真实请求里不够稳定，也会额外消耗次数
- 因此当前版本默认不输出“行李规则 / 退改签”占位文案；拿不到就直接省略

### 5.4 怎么搜更容易找到最便宜的票

想找“节假日里真正便宜的票”，建议这样配：

1. 不要一上来就固定死单一去返日期  
   先给一个日期窗口，让系统在窗口里挑组合。

2. 节假日场景优先用“窗口 + 约束”，不要只查一天  
   当前这套规则已经会自动要求：
   - 去返日期完整覆盖端午假期
   - 总跨度不超过 6 天
   - 假期外请假工作日不超过 3 天

3. 第一次先缩小城市和窗口，确认链路通了再放大  
   例如先配：
   - `origins: [HKG]`
   - `destination: PQC`
   - `thailand_destinations: [BKK]`
   - `window_start/window_end`: 5 到 7 天

4. 真正想比价格时，再逐步放大窗口  
   例如从 6 天窗口扩大到 10 天窗口，而不是第一次就把所有城市和所有目的地一起扫。

5. 如果你已经知道大致出行周末  
   就配 `fixed_depart_date/fixed_return_date`，让系统围绕这组日期扩窗，而不是完全盲扫。

## 6. 历史位置判定规则

对同一条监控维度（出发地、目的地、去返日期、provider、source_currency）统计历史价格：

- 新数据不足：`新样本`
- 若当前价格接近历史低位区间：`低位`
- 若当前价格接近历史高位区间：`高位`
- 其余：`中位`

建议至少运行多轮后再参考位置判断。

## 7. 常见问题

### 7.1 抓取偶发失败

表现：`[WARN] Trip 页面未解析到价格 ...`

处理：

- 重试运行（已内置重试）
- 调大 `trip_scrape_timeout_seconds`
- 网络稳定后再跑

### 7.2 没有航班号

部分页面/会话不会稳定暴露航班号，当前策略会尝试二跳详情抓取；若仍不可见则输出 `N/A`。

### 7.3 看不到行李规则或退改签

这不一定是程序错误，常见原因有：

- 当前 Google Flights 搜索结果本身不稳定提供这些字段
- 相关信息往往只在更后面的 booking options / 下单页里才完整出现
- 这条链路在真实请求里不够稳定，且会额外消耗 API 次数

当前策略是：

- 当前版本直接省略这两类字段
- 不再显示误导性的占位文案

### 7.4 Google Flights 配额用尽怎么办

如果你用的是：

```yaml
provider: google_flights
```

当 Google Flights 因以下原因不可用时：

- 免费额度用尽
- API 返回错误
- 网络请求失败

程序会在当前这轮运行里自动切换到 `trip_scrape`，继续完成后面的查询，不需要手动改配置。

你会在日志里看到类似：

```text
[PROVIDER] Google Flights 不可用，自动切换到 trip_scrape: ...
```
